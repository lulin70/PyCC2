## 二、TD-072 enhanced_sound_bridge.py 拆分方案

### 2.1 现状

- **文件**: `src/pycc2/presentation/audio/enhanced_sound_bridge.py` (949L)
- **类**: `EnhancedSoundSystem` (@dataclass, `__init__(self)` 无参)
- **TD-067 评估**: ✅ TRUE God Class — 单类承担"音频桥接"+"程序化波形合成"两个不相干职责
- **模块级**: `CombatSoundEvent` 枚举 (29 成员) + `SoundFileMapping` dataclass + `DEFAULT_SOUND_MAPPINGS` (15 项) + `_enhanced_sound_instance` 单例 + `get_enhanced_sound_system()` 工厂

### 2.2 职责分组

**职责 A: 音频桥接（系统侧，保留）** ~11 方法
- 生命周期: `__init__` / `_initialize_default_mappings` / `initialize` / `shutdown`
- 缓存/加载: `_preload_sounds` / `_cache_sound` / `_get_cached_sound` / `_load_sound_from_file` / `register_sound_mapping`
- 派发: `play_combat_event` (facade 入口，先查缓存→文件→合成器)
- 属性: `master_volume` / `sfx_volume` (getter+setter, 0-1 边界)
- 17 个 `play_*` 便捷方法 (facade, 委托 `play_combat_event`)

**职责 B: 程序化波形合成（提取）** ~15 方法 ~500 LOC
- `_generate_cc2_combat_fallback` (DSP 派发)
- `_generate_procedural_fallback` (DSP 派发 + 委托 `sound_system.ProceduralSoundGenerator`)
- 13 个 `_gen_*`: tank_cannon_fire / at_rocket_fire / mortar_launch / grenade_explosion_short / airstrike_bomb / vehicle_engine(state) / armor_penetrate / ricochet_bounce / near_miss_whizz / smoke_deploy_hiss / suppression_fire
- **特征**: 仅读 `self._sfx_volume`，返回 `np.ndarray`，与缓存/加载零协作 → 无状态

### 2.3 拆分模式: facade + 子模块（组合委托）

**为何不用 mixin 模式**（D11 sprite_renderer.py 先例）:
1. 合成器方法是**无状态**的（仅读标量 `_sfx_volume`，返回 ndarray），不像 sprite mixin 共享可变渲染状态
2. TD-072 明确要求"组合委托"
3. 合成器无 `self` 状态依赖，纯函数式更清晰

**产出**:

```
src/pycc2/presentation/audio/
├── enhanced_sound_bridge.py        # 保留: EnhancedSoundSystem (组合委托)
├── procedural_sound_synthesizer.py # 新建: ProceduralSoundSynthesizer (15 个合成方法)
└── sound_system.py                 # 不变 (已有 ProceduralSoundGenerator 类，避免重名)
```

**新类 `ProceduralSoundSynthesizer`**:
- 构造: `__init__(self, sfx_volume: float = 0.8)`
- 公共方法: `generate_for(event: CombatSoundEvent) -> np.ndarray | None` (派发入口)
- 私有方法: 15 个 `_gen_*` + 2 个 `_generate_*_fallback`
- 状态: 仅 `self._sfx_volume: float`
- numpy 在每个 `_gen_*` 内 lazy import (保持原模式)

**修改 `EnhancedSoundSystem`**:
- `__init__` 内 `self._synth = ProceduralSoundSynthesizer(self._sfx_volume)`
- `_generate_procedural_fallback` 改为 `return self._synth.generate_for(event)`
- `_generate_cc2_combat_fallback` 同上
- `sfx_volume` setter 同步 `self._synth._sfx_volume = value`
- **清理**: 移除 `play_combat_event` 的 `position` 参数 docstring 承诺 (3D positional audio 未实现)

### 2.4 Public API 兼容性

**100% 向后兼容**:
- `EnhancedSoundSystem` 类名 / `__init__(self)` 签名 / 所有 public 方法签名不变
- `CombatSoundEvent` / `SoundFileMapping` / `DEFAULT_SOUND_MAPPINGS` / `get_enhanced_sound_system` 模块级导出不变
- `from pycc2.presentation.audio.enhanced_sound_bridge import *` 行为不变

**测试影响**:
- `tests/unit/test_phase_b_final_sprint.py:806-921` TestEnhancedSoundBridge (10 tests) — 无需修改导入
- `tests/unit/test_smoke_zero_coverage.py:277-284` — 无需修改导入
- 可选: 为 `ProceduralSoundSynthesizer` 新增直接单测（覆盖 `_gen_*` DSP 路径，当前未覆盖）

### 2.5 风险与缓解

| 风险 | 缓解 |
|------|------|
| 拆分后合成器行为变化 | 保持 numpy lazy import 模式 + 所有 `_gen_*` 实现原样迁移 |
| `sfx_volume` 同步问题 | setter 双向同步，单测验证 |
| 重名冲突 | 新类名 `ProceduralSoundSynthesizer` ≠ 既有 `ProceduralSoundGenerator` (sound_system.py) |
| 测试覆盖不足 | 拆分后为 ProceduralSoundSynthesizer 补 smoke 测试 |

## 四、3 God Class 候选评估结论

### 4.1 评估结果

| 文件 | 行数 | 方法数 | 真实 God Class? | 处置 |
|------|------|--------|-----------------|------|
| `deployment_ui.py` | 689 | 50 | ❌ FALSE | 保留 — Facade 模式，已提取 7 个协作者 |
| `sound_system.py` | 741 | 43 | ❌ FALSE | 保留 — 单一内聚域"音频引擎" |
| `sprite_renderer_base.py` | 303 | 44 | ❌ FALSE | 保留 — D11-2 SRP 拆分产物，按设计 Facade base |

### 4.2 FALSE 判断理由

**deployment_ui.py (50 方法)**: v0.3.29-v0.3.32 已完成 SRP 重构，提取 7 个协作者类 (DeploymentDragDrop / DeploymentOrders / DeploymentLOSSystem / DeploymentRenderer / DeploymentZoneBuilder / DeploymentPlacementService / DeploymentInputRouter)。50 方法中绝大多数是 1-2 行委托包装 (如 `return self._placement.place_unit(...)`)。**Facade 模式**，单一职责"部署 UI 协调"。

**sound_system.py (43 方法)**: 文件含 6 个类，已多次拆分。最不相干的职责 (DSP 波形合成) 已提取到 `ProceduralSoundGenerator` (12 个 `generate_*` classmethod，纯 numpy，无状态)；`MusicPlayer` 已分离。`SoundSystem` 剩余职责 (混音器生命周期 / 缓存 / 播放 / 音量 / 空间音频 / 优先级调度 / 音乐 ducking) 均为"音频引擎"内聚子方面，共享 `_cache`/`_active_sounds`/`_config`/`_mixer_config` 状态，调用链紧密。**单一内聚域**，与 enhanced_sound_bridge (桥接+DSP 混合两个完全不相干职责) 有本质区别。

**sprite_renderer_base.py (44 方法)**: D11-2 SRP 拆分产物，明确设计为 **Base/Facade 协调器**。44 方法中: 12 个 `@property` 是 backward-compat shim (委托 `_cache_manager`/`_effect_renderer`)；6 个 `spawn_*` 是 1 行委托；4 个 `update_weather/shell_casings/suppression_overlay/_smooth_positions` 是显式 no-op (系统已迁至专用 renderer)；`render()` 是 orchestrator。继承自 4 个 focused rendering mixin (terrain/vl-flag/unit/unit-overlay)。**按设计非 God Class**，是 Facade base。

### 4.3 D13 教训再次验证

D13 N-1 用"方法数 >30"机械阈值列出候选 → D14/v0.4.5 复核大部分为 FALSE。本次评估剩余 3 个候选全部 FALSE。**累计 3 批次 12 候选 → 1 TRUE / 11 FALSE → 8.3% TRUE hit rate (91.7% 误判率)**。God Class 评估必须基于"单类多职责"而非方法数/行数机械阈值。
