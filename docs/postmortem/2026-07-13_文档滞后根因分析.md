## 3. 5-Why 根因分析

### Why 1: 为什么 PRD/DESIGN 没有随版本更新？

**回答**：因为版本发布流程中没有"文档一致性检查"步骤。每次发布只检查 VERSION/README/ROADMAP/CHANGELOG，不检查 PRD/DESIGN。

### Why 2: 为什么发布流程不检查 PRD/DESIGN？

**回答**：因为 DevSquad 的 "Delivery Workflow Iron Rules" Step 5（Docs Update）列出了"必须检查的文档清单"，但这个清单是建议性的，没有被强制执行。PRD/DESIGN 在清单中，但没有人逐项核对。

### Why 3: 为什么清单没有被强制执行？

**回答**：因为缺乏自动化检查机制。CI 只检查代码质量（ruff/mypy/bandit/pytest），不检查文档版本一致性。人工检查容易遗漏。

### Why 4: 为什么 CI 不检查文档版本一致性？

**回答**：因为项目将"功能交付"优先于"文档同步"。CI 设计目标是"确保代码可发布"，而非"确保文档与代码一致"。文档一致性被视为"非阻塞"项。

### Why 5: 为什么文档一致性被视为"非阻塞"？

**回答**：**这是根本原因**——存在认知偏差：将文档视为"一次性产出"而非"持续维护的活文档"。具体表现：

1. **文档分层认知缺失**：VERSION/README 是"对外门面"（每次更新），PRD/DESIGN 是"内部设计"（一次性写完就不管），但这种分层是错误的——PRD/DESIGN 是决策依据，过期会导致错误决策
2. **"代码即文档"误区**：认为代码注释和 docstring 替代了设计文档，但 PRD/DESIGN 承载的是"为什么"和"做什么"，代码承载的是"怎么做"
3. **反馈循环缺失**：v0.4.16 的诚实修正（88%→75%）更新了 ROADMAP，但没有触发 PRD 更新，说明文档间没有"联动更新"机制

### 4.1 流程层面

### 4.2 工具层面

## 5. 改进建议

### 5.1 短期（立即可做）

**S1: 在 release.yml 中添加文档版本一致性检查**

```yaml
- name: Verify document version consistency
  run: |
    VERSION=$(cat VERSION)
    for doc in docs/PRD.md docs/DESIGN.md docs/ROADMAP.md docs/PROJECT_STATUS.md; do
      if ! grep -q "v$VERSION\|$VERSION" "$doc"; then
        echo "::error::$doc does not reference version $VERSION"
        exit 1
      fi
    done
```

**S2: 在 Delivery Workflow Step 5 中强制检查 PRD/DESIGN**

在 IMPROVEMENT_PLAN_V070.md 的 Wave 1 文档校准清单中添加 PRD/DESIGN。

### 5.2 中期（1-2 周内）

**M1: 创建文档一致性 CI 检查脚本**

```bash
#!/bin/bash
# scripts/check_doc_consistency.sh
VERSION=$(cat VERSION)
ERRORS=0
for doc in docs/PRD.md docs/DESIGN.md docs/ROADMAP.md docs/PROJECT_STATUS.md docs/TECH_DEBT.md docs/TEST_PLAN.md; do
  if ! grep -qE "v?${VERSION}" "$doc"; then
    echo "ERROR: $doc does not reference version $VERSION"
    ERRORS=$((ERRORS+1))
  fi
done
exit $ERRORS
```

在 ci.yml lint job 中调用此脚本。

**M2: 建立文档 owner 制度**

| 文档 | Owner | 更新频率 |
|------|-------|----------|
| PRD.md | PM 角色 | 每次 MINOR 版本 |
| DESIGN.md | Architect 角色 | 每次架构变更 |
| ROADMAP.md | PM 角色 | 每次版本发布 |
| TECH_DEBT.md | Coder 角色 | 每次技术债变更 |

### 5.3 长期（持续改进）

**L1: 文档即代码（Docs as Code）**

将文档版本号与代码版本号强绑定，CI 自动检查所有 `.md` 文件中的版本号一致性。

**L2: 文档变更触发器**

在 git hooks 或 CI 中添加检查：如果 `src/` 有变更但 `docs/` 无变更，发出警告（"代码变更但文档未更新"）。

**L3: 文档健康度仪表盘**

在 PROJECT_STATUS.md 中添加"文档健康度"指标：
- 版本一致性：所有文档版本号 == VERSION 文件
- 内容时效性：文档最后更新日期 vs 代码最后更新日期
- 链接有效性：文档间引用链接是否有效

## 7. 教训总结

> **核心教训**：文档一致性不是"一次性校准"，而是"持续维护"。每次版本发布时，必须检查所有文档（不仅仅是 README/CHANGELOG）的版本号和内容时效性。

> **认知修正**：PRD/DESIGN 不是"一次性设计文档"，而是"活的设计资产"。它们承载的不仅仅是"初始设计"，还有"演进历史"和"当前状态"。如果它们过期，新贡献者和 AI 都会基于错误信息做决策。

> **流程修正**：DevSquad 的 "Meta Iron Rule: 文档先行" 需要扩展为 "文档先行 + 文档同步"。不仅要"写代码前先写文档"，还要"改代码后同步文档"。

> **工具修正**：CI 不仅要检查代码质量，还要检查文档一致性。文档版本号是"文档健康度"的最基本指标。
